01 What Is This Project? 01
The Business
Lola's Party System is a real party rental business in Woodbridge, Virginia, owned by Nora Cortes ("Ms. Cortes"). She rents out soft play packages, table setups, mini bouncers, and handmade party favors for children's birthday parties and events.

03 Google Cloud Functions — The Backend 03
What Is a Cloud Function?
A Cloud Function is a single Python function that runs in Google's cloud. You don't manage a server — Google handles all of that. You just write a function, deploy it, and Google gives you a URL. When someone visits that URL (or sends data to it), your function runs and sends back a response.
Think of it like a vending machine

You put money in (send a request with data), press a button (the function runs), and a product comes out (you get a response). You don't need to know how the machine works inside. Each Cloud Function is a separate vending machine that does one specific thing.

Every Cloud Function in our project follows this pattern:
main.py — anatomy of a cloud function # 1. The decorator tells Google "this is a Cloud Function" @functions_framework.http def my_function(request): # 2. Handle CORS (allow the browser to talk to this function) cors_headers = build_cors_headers(request) if request.method == 'OPTIONS': return ('', 204, cors_headers) # 3. Read the data the frontend sent data = request.get_json(silent=True) # 4. Do the work (database reads, calculations, API calls) result = do_something(data) # 5. Send back a response (JSON + status code + CORS headers) return ({"status": "success"}, 200, cors_headers)
Every single function in main.py follows this exact 5-step pattern. Once you understand this pattern, you understand every endpoint.

04 Firestore Database — Where Data Lives 04
What Is Firestore?
Firestore is a NoSQL document database by Google. Unlike a traditional SQL database with rows and columns (like Excel), Firestore stores data as documents inside collections.
Think of it like a filing cabinet

A collection is a drawer (e.g., "bookings"). Each document is a folder inside that drawer (e.g., one booking). Inside each folder are fields — labeled pieces of data (customer name, date, total price, status). Every document has a unique ID, like a folder label.

how firestore data is structured # Collection: "bookings" # Document ID: "abc123" (auto-generated) # Fields: { "customer": { "name": "Sarah Johnson", "email": "sarah@email.com", "phone": "(703) 555-1234" }, "date": "2026-04-15", "time": "14:00", "status": "Pending Approval", "payment_status": "Unpaid", "total_price": 530.00, "items": [{"id": "SP-001", "child_name": "Emma"}], "delivery_address": "123 Main St, Woodbridge VA", "created_at": /* server timestamp */ }

05 APIs — How Frontend Talks to Backend 05
What Is an API?
API stands for Application Programming Interface. It is a way for two programs to talk to each other. In our project, the frontend (JavaScript in the browser) talks to the backend (Python Cloud Functions) through APIs.
When the frontend needs something from the backend, it sends an HTTP request to a URL. The backend processes the request and sends back an HTTP response with data (usually in JSON format).
Think of it like ordering at a drive-through

You (the frontend) drive up to the speaker (the API URL) and place your order (send a request with data). The kitchen (the Cloud Function) makes your food (processes the request) and hands it to you through the window (sends back a response). You never go into the kitchen yourself.

frontend — javascript fetch call (from inquire.html) // The frontend sends data to the Cloud Function URL const response = await fetch('https://-.cloudfunctions.net/check_availability', { method: 'POST', // POST = sending data headers: { 'Content-Type': 'application/json' }, // "I'm sending JSON" body: JSON.stringify({ // The data we're sending date: '2026-04-15', items: [{id: 'SP-001'}], address: '123 Main St, Springfield VA' }) }); const data = await response.json(); // Parse the response // data = { available: true, quote: { subtotal: 400, delivery: 100, tax: 30, total: 530 } }
Request and Response
Every API interaction has two parts:
Request (Frontend Sends)
URL: Which Cloud Function to call
Method: POST (send data) or GET (fetch data)
Headers: Metadata (content type, auth token)
Body: The actual data (JSON)
Response (Backend Returns)
Status Code: 200 (success), 400 (bad request), 401 (unauthorized), 500 (server error)
Body: The result data (JSON)
Headers: CORS headers (see below)
backend — how a cloud function returns a response # Every return is a tuple: (data, status_code, headers) return ( {"available": True, "quote": {"total": 530.00}}, # JSON data 200, # Status: OK cors_headers # CORS (explained below) )

CORS — Why Browsers Block Requests
CORS (Cross-Origin Resource Sharing) is a browser security feature. When your website at .web.app tries to call a Cloud Function at -.cloudfunctions.net, the browser says: "Wait, those are different domains. I need permission."
The backend must include special headers in its response saying "yes, this origin is allowed." That is what build_cors_headers() does:
main.py — cors configuration ALLOWED_ORIGINS = { "https://.web.app", "https://.firebaseapp.com", } def build_cors_headers(request): origin = request.headers.get("Origin", "") headers = { "Access-Control-Allow-Methods": "POST, OPTIONS, GET", "Access-Control-Allow-Headers": "Content-Type, Authorization", } if origin in ALLOWED_ORIGINS: headers["Access-Control-Allow-Origin"] = origin # "Yes, you're allowed" return headers
Browsers also send a preflight request (an OPTIONS request) before the actual request to check if the server allows it. That is why every Cloud Function starts with:
if request.method == 'OPTIONS': return ('', 204, cors_headers) # 204 = "OK, no content, just headers"
Why only two origins? We only allow requests from our own website domains. A random website at evil-site.com cannot call our Cloud Functions from a browser because its origin is not in our allowlist.

Webhooks — How Stripe Tells Us "Payment Received"
After a customer pays, we cannot just trust the browser redirect to success.html — the customer could have closed the tab or faked the URL. Instead, Stripe sends a webhook — a POST request from Stripe's servers directly to our stripe_webhook Cloud Function.
Think of it like a delivery confirmation

When you order a package, the delivery driver doesn't just say "delivered" — the shipping company (Stripe) sends a verified notification directly to you (our webhook function). This is trustworthy because it comes from Stripe, not from the customer.

Google Maps — Delivery Fee Calculation
Delivery fees are calculated using the Google Maps Distance Matrix API. It measures the actual driving distance (not a straight line) between the customer's address and Woodbridge, VA.
main.py — delivery fee logic LOLA_HQ_ADDRESS = "Woodbridge, VA" BASE_DELIVERY_FEE = 100.00 # $100 base fee FREE_RADIUS_MILES = 30.0 # Free within 30 miles COST_PER_EXTRA_MILE = 1.25 # $1.25 per mile beyond 30 def get_delivery_fee(address): # Call Google Maps to get actual driving distance matrix = gmaps_client.distance_matrix( origins=LOLA_HQ_ADDRESS, destinations=address, units='imperial' ) meters = matrix['rows'][0]['elements'][0]['distance']['value'] miles = meters * 0.000621371 if miles > FREE_RADIUS_MILES: extra = (miles - FREE_RADIUS_MILES) * COST_PER_EXTRA_MILE return BASE_DELIVERY_FEE + extra return BASE_DELIVERY_FEE # Within 30 miles = flat $100
Example: A customer 45 miles away pays $100 + (15 extra miles * $1.25) = $118.75 delivery. A customer 20 miles away pays the flat $100.

Firestore Security Rules
Firestore has its own security layer — rules that control who can read and write data directly. These rules are defined in firestore.rules and deployed with firebase deploy --only firestore.
firestore.rules — our security rules // Bookings: ONLY admin can read/write from the browser match /bookings/{bookingId} { allow read, write: if request.auth != null && request.auth.uid == 'RckCH5X...'; // Admin UID } // Inventory: ANYONE can read (prices shown to customers) // Only admin can write (toggle items on/off) match /inventory/{itemId} { allow read: if true; allow write: if request.auth.uid == 'RckCH5X...'; } // Gallery: ANYONE can read, only admin can write match /gallery/{docId} { allow read: if true; allow write: if request.auth.uid == 'RckCH5X...'; } // Everything else: DENY by default match /{document=**} { allow read, write: if false; }
Important distinction: Cloud Functions bypass Firestore security rules because they use the Admin SDK (server-side). The rules only apply to client-side access (JavaScript in the browser). That is why submit_inquiry (a Cloud Function) can write to bookings, but a random person with the Firebase config cannot.

2 How Origin-Based Routing Works 2
The Routing Flow
Every request to a Cloud Function includes an Origin header that identifies which site sent it. The backend uses this to pick the right Firestore database.
Database Routing Flow
Request arrives
from browser or scheduler
get_db(request)
checks Origin header
Test origin? → -test DB
Prod origin? → (default) DB
No origin? → (default) DB (cron jobs, Stripe)
The routing logic lives in helpers.py:
helpers.py — Origin-Based Database Routing
db = firestore.Client()                                    # Production database
db_test = firestore.Client(database='-test') # Test database

TEST_ORIGINS = {
    "https://-test.web.app",
    "https://-test.firebaseapp.com",
    "http://localhost:5000",
    "http://127.0.0.1:5000",
}

def get_db(request):
    origin = request.headers.get("Origin", "")
    if origin in TEST_ORIGINS:
        return db_test
    return db
Important: Cloud Scheduler cron jobs and Stripe webhooks have no Origin header, so they always route to production. This is intentional — automated processes should always work with real data.

Backup & Restore Guide

in Lola's Party System

A practical guide to the automated daily backup system. Learn what gets backed up, how the backup Cloud Function works, and exactly how to restore data if disaster strikes.

5 Sections 5 Concepts Ready-to-Run Commands
1 The Backup Policy — What, When, How Long 1
Daily Automated Backups
Every production system needs a backup strategy. If data gets accidentally deleted, corrupted, or a bad deploy wipes something out, you need a way to recover. Lola's Party System runs automated backups every night with a 7-day retention window.
SettingValueWhy
FrequencyDaily at 3:00 AM ETOff-peak hours — no customer traffic, captures the day's final state
Retention7 days rollingEnough time to catch accidental deletions; old backups auto-deleted to control costs
Backup Bucketgs://-backupsDedicated bucket in the US multi-region (required by Firestore export API)
Lifecycle PolicyDelete objects older than 7 daysWithout this, backups accumulate forever — after 1 year you'd have 365 copies of mostly the same data
Restore MethodManual via gcloud / gsutilRestore is rare; a documented runbook is simpler and safer than an automated button
Estimated Cost~$0.50–2/month7 snapshots × ~50 MB each = ~350 MB steady state
Think of it like a security camera

The backup runs automatically every night, like a security camera that records on a loop. You hope you never need the footage, but if something goes wrong, you can rewind to any of the last 7 days. After 7 days, old tapes are recorded over to save space — this is what the lifecycle policy does automatically.

Why 7 days? For a small rental business processing 5–15 bookings per week, 7 days is plenty of time to notice something went wrong. If you haven't caught an issue within a week, a backup probably wouldn't help anyway — too much new data has accumulated. You can always increase retention by changing the lifecycle policy on the bucket.
2 What Gets Backed Up — And What Doesn't 2
Backup Coverage Map
Not everything needs a backup. Some data is already protected by the services that manage it. Here is the complete breakdown of what is and isn't backed up, and why:
DataBacked Up?MethodWhy
Firestore: bookings Yes Firestore managed export Revenue-critical — customer orders, payment status, contact info
Firestore: inventory Yes Firestore managed export Product catalog — package names, prices, availability flags
Firestore: gallery Yes Firestore managed export Gallery metadata — photo URLs, categories, sort order
Cloud Storage photos
6 folders: party-favors, softplay, about, tables, mini-bouncers, gallery		 Yes Blob copy to backup bucket Product images and gallery photos — hard to recreate if lost
Google Sheets ledger No — Google Sheets has built-in version history (30+ days) — already protected
Stripe payment data No — Stripe maintains complete records in their dashboard — already protected

Secrets / API keys No — Never store secrets in backups — regenerate from Stripe, Google, and Firebase consoles
Test database No — Test data is disposable — can be re-seeded from production at any time
Collection-level vs. document-level backups: Firestore managed exports back up an entire collection at once. To restore, you import the whole collection back — you cannot restore a single document from a managed export. For a small business app, re-entering one lost booking from the confirmation email is faster than building and maintaining a document-level restore system.
3 How the Backup Function Works 3
Cloud Function: cron_backup_firestore
The backup is powered by Cloud Function #14 (cron_backup_firestore), triggered daily at 3:00 AM ET by Cloud Scheduler. Here is the complete flow:

The function performs two operations in sequence:
Step 1: Firestore Managed Export
Calls the Firestore Admin REST API (exportDocuments) to export all 3 collections to the backup bucket. This is an asynchronous operation — the API returns immediately with an operation ID, and the export continues in GCP's backend.
main.py — Firestore Export (simplified)
export_url = f"https://firestore.googleapis.com/v1/projects/{PROJECT_ID}/databases/(default):exportDocuments"
export_body = {
    "outputUriPrefix": f"gs://{BACKUP_BUCKET}/firestore/{today_str}",
    "collectionIds": ["bookings", "inventory", "gallery"]
}

# Uses service account credentials with datastore scope
credentials, _ = google.auth.default(scopes=["https://www.googleapis.com/auth/datastore"])
# POST to Firestore Admin API — returns immediately, export runs in background
resp = urllib.request.urlopen(req, timeout=120)
Step 2: Cloud Storage Photo Copy
Copies all photos from 6 folders in the source bucket to a date-stamped folder in the backup bucket. This is synchronous — each blob is copied one at a time.
main.py — Photo Sync (simplified)
PHOTO_FOLDERS = ["party-favors", "softplay", "about", "tables", "mini-bouncers", "gallery"]

for folder in PHOTO_FOLDERS:
    blobs = src_bucket.list_blobs(prefix=f"{folder}/")
    for blob in blobs:
        dest_name = f"storage-photos/{today_str}/{blob.name}"
        src_bucket.copy_blob(blob, dst_bucket, new_name=dest_name)
        photos_copied += 1
Why 3:00 AM? No customer traffic at that hour. The backup runs after the other cron jobs (cron_auto_complete_orders and cron_check_balances) finish, so it captures the day's final state including any automated status changes.
Failure alerts: If the backup fails for any reason (permissions, bucket issues, timeout), the function sends an alert email to your-business-email@example.com with the error details and a timestamp. Cloud Scheduler also retries automatically on 5xx responses.
4 Restore Runbook — When Disaster Strikes 4
Four Restore Scenarios
If disaster strikes, follow the appropriate scenario below. Replace YYYY-MM-DD with the backup date you want to restore from (e.g., 2026-03-27). You can list available backups with: gcloud storage ls gs://-backups/firestore/
Scenario A — Restore Entire Firestore Database
When to use: Multiple collections are corrupted, a major incident occurred, or you need a full rollback.
Restore all 3 collections (bookings + inventory + gallery)
gcloud firestore import gs://-backups/firestore/YYYY-MM-DD/
This overwrites ALL current data in bookings, inventory, and gallery with the backup snapshot. Any bookings created after the backup date will be lost. Make sure this is what you want.
Scenario B — Restore a Specific Collection
When to use: Only one collection is affected (e.g., someone accidentally deleted bookings, but inventory and gallery are fine).
Restore only the bookings collection
gcloud firestore import gs://-backups/firestore/YYYY-MM-DD/ \
  --collection-ids=bookings
Valid collection IDs: bookings, inventory, gallery. You can restore multiple: --collection-ids=bookings,inventory
Scenario C — Restore Cloud Storage Photos
When to use: Product photos or gallery images were accidentally deleted from Cloud Storage.
Restore all photos from a specific backup date
gsutil -m rsync -r gs://-backups/storage-photos/YYYY-MM-DD/ \
  gs:///
The -m flag enables multi-threaded transfer for faster restore. The -r flag recurses into all 6 photo folders.
Scenario D — Lost Secrets / Credentials
When to use: API keys or passwords are compromised or lost. Secrets are never stored in backups — regenerate from the source.
SecretWhere to Regenerate
STRIPE_SECRET_KEYStripe Dashboard → Developers → API Keys
STRIPE_WEBHOOK_SECRETStripe Dashboard → Developers → Webhooks
EMAIL_PASSGoogle Account → Security → App Passwords

Post-Restore Verification Checklist

After any restore, confirm all of these work before considering it resolved:
  • Admin dashboard loads and displays bookings correctly
  • Inquiry form submits successfully (creates a new test booking)
  • Party favors page displays product photos from Cloud Storage
  • Gallery page displays uploaded photos
